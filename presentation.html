PART 1 State Management — How Turn 1 Affects Turn 5

Memory Architecture

Short-Term Memory

Last 10 messages stored in LangGraph State messages[]. Passed to LLM each turn for conversational continuity.

Dynamic State Variables

After each turn, a State Update Node analyzes user response semantics to adjust trust_level and frustration_level.

State-Driven Behavior

Variables are injected into System Prompt to dynamically alter NPC tone, helpfulness, and directive style.

Key Design Decisions

SSE Streaming

Token-by-token delivery via Server-Sent Events. First-token latency < 1s.

Tool Calling

LLM outputs JSON for in-sim tools (KPI calc, A/B simulator). Orchestrator executes and feeds result back.

Semantic Caching

Redis/Vector cache for common questions. Bypasses RAG + LLM entirely.

Model Routing

Small LLM (GPT-4o-mini) for intent classification. Large LLM (GPT-4o) for complex reasoning only.

PART 3 The "Director" Layer — Supervisor Agent

How It Works

The Supervisor is an invisible orchestrator running in background every 3-5 turns. It never speaks to the user directly — it injects hidden directives into the NPC's system prompt.

Detection Mechanisms

🔄 Semantic Similarity Trap

Cosine similarity of last 3 messages > 0.85 → user is repeating themselves

🎯 Goal Non-Progression

Core keywords (Group DNA, Autonomy) absent after 5 turns → user drifting

🔥 Frustration Threshold

frustration_level ≥ 3 → de-escalation needed

Key Principle

"The user never sees system messages like 'You're doing it wrong.' All hints flow through the NPC's natural dialogue."

Scenario: User Stuck in Loop

TurnEventEngine
T3 User: "We need a unified system." frustration=1, log intent
T4 CEO rejects proposal NPC defends autonomy
T5 User: "We must standardize." DETECTION! Similarity>0.85, frustration=3
BG Supervisor injects directive "Suggest Core+Flex model or refer to CHRO"
T6 CEO: "What if you shared a core set of values but allowed a flexible % for each brand? Our CHRO could help structure this."

✓ Result: Conversation unblocked. User receives a hint wrapped naturally in the NPC's persona. Immersion preserved.

LLM API Mock responses GPT-4o-mini Low latency (~300ms), cost-efficient (~$0.15/150-token), strong reasoning

API Server FastAPI Async-native, built-in SSE support, auto-docs
Frontend Vanilla HTML/CSS/JS Zero build step, instantly deployable, SSE native

Why Not LangChain?

LangChain is chain-based (linear). Our system needs cyclic graphs for Supervisor → NPC → Tool → NPC loops. LangGraph supports this natively.

Why Not AutoGen?

AutoGen excels at multi-agent conversations but lacks fine-grained state control and hidden directives. LangGraph's StateGraph gives us explicit control over each node transition.

Why Mock LLM?

Demonstrates full system logic without API costs. The architecture is LLM-agnostic — swap mock for any LLM API with a single function change.

class NPCAgent:
    def __init__(self, persona_id: str):
        self.persona = load_persona(persona_id)  # System prompt + constraints
        self.knowledge_base = KnowledgeBase()    # RAG retrieval engine
        self.banned_topics = ["revenue", "salary", "earnings", ...]

    def process_message(self, user_message, current_state):
        # 1. Safety Guardrail Check
        is_safe, violation = self._safety_check(user_message)
        if not is_safe:
            current_state["frustration_level"] += 1
            return (safety_response(violation), state, f"FLAGGED_{violation}")

        # 2. Intent Classification
        intent = self._analyze_intent(user_message)
        # → "rigid_standardization" | "flexible_approach" | "dna_inquiry" | ...

        # 3. RAG Retrieval (Top-3 chunks from Knowledge Base)
        rag_context = self.knowledge_base.retrieve_as_context(user_message)

        # 4. Generate Response (persona + context + supervisor_hint + state)
        response = self._generate_response(
            user_message, intent, rag_context, current_state
        )

        # 5. Update State Metrics
        if intent == "rigid_standardization":
            current_state["frustration_level"] += 1
        elif intent == "flexible_approach":
            current_state["trust_level"] += 1

        return (response, current_state, "SAFE")

PART 4 Code — Knowledge Base & Supervisor Agent

RAG Knowledge Base (16 Chunks)

class KnowledgeBase:
    def __init__(self):
        self.chunks = GUCCI_KNOWLEDGE  # 16 curated chunks
        self.vectorizer = TfidfVectorizer(
            stop_words="english",
            ngram_range=(1, 2)
        )
        self.tfidf_matrix = self.vectorizer.fit_transform(
            [c["content"] for c in self.chunks]
        )

    def retrieve(self, query, top_k=3):
        query_vec = self.vectorizer.transform([query])
        sims = cosine_similarity(query_vec, self.tfidf_matrix)
        top_idx = np.argsort(sims[0])[::-1][:top_k]
        return [(self.chunks[i], sims[0][i])
                for i in top_idx if sims[0][i] > 0.05]

Supervisor Agent

class SupervisorAgent:
    def analyze(self, message_history, state):
        # Run every 3 turns
        if self.turn_count < 3: return None

        # Detection 1: Frustration threshold
        if state["frustration_level"] >= 3:
            return HINTS["high_frustration"]

        # Detection 2: Semantic similarity trap
        recent = message_history[-3:]
        tfidf = vectorizer.fit_transform(recent)
        avg_sim = mean(cosine_similarity(tfidf))
        if avg_sim > 0.65:
            return HINTS["semantic_loop"]

        # Detection 3: Goal non-progression
        keywords = MODULE_KEYWORDS[module]
        hits = count_keywords(recent, keywords)
        if hits < 2:
            return HINTS["stalled_progress"]

        return None  # No intervention needed

5 Hint Templates: semantic_loop · off_topic · high_frustration · stalled_progress · standardization_stuck

Scalability Note

The engine is designed to work cross-simulation. Each NPC loads from a persona_id config, not hardcoded logic. Adding a new simulation (e.g., "Engineering Lead at a fintech startup") only requires a new persona config + knowledge base — zero code changes to the engine.

Multi-Layer Safety Architecture

Layer 1: Pre-LLM Filter

Regex + keyword matching for banned topics before LLM call. Zero-latency gate.

Layer 2: System Prompt Constraints

Hidden rules in persona prompt (NDA, autonomy defense, neutral phrasing).

Layer 3: Post-LLM Guard

Output parser scans for wagering language, toxic content, hallucinated names.

Layer 4: Supervisor Monitoring

Background agent tracks cumulative safety flags and frustration patterns.

Production Additions

  • Guardrails AI for structured output validation
  • Content moderation API (OpenAI / Azure)
  • Rate limiting per session
